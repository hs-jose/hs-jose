异常、断言和日志

程序发生错误时

  1. 向用户通知错误
  2. 保存所有工作
  3. 允许用户妥善的退出程序

断言有选择的启用检查

日志记录出现的问题

处理错误

程序出现错误而使得某些操作无法完成时

  • 允许返回一种安全状态，并能让用户执行其他命令
  • 或允许用户保存所有工作结果，并以妥善的方式终止程序

Java 使用异常处理 (exception handing) 的错误捕获机制

​ 异常处理的任务是将控制权从产生错误的地方转移到能够处理这种情况的错误处理器

可能出现错误的问题

  • 用户输入错误
  • 设备错误
  • 物理限制
  • 代码错误

如果方法不能够以正确的途径完成任务，可以抛出 (throw) 一个封装了错误信息的对象。这个方法会立刻退出，并不返回正常值 (或任何值)。同时异常处理机制开始搜索能够处理这种异常状况的异常处理器 (exception handler)

异常分类

所有异常都是派生于 Throwable 类的一个类实例

  • Throwable
    • Error
    • Exception
      • 其他异常 (例如 IOException)
      • RuntimeException

java异常层次结构简化示意图

Error 类层描述 Java 运行时，系统的内部错误和资源耗尽错误。我们自身的程序不应该抛出这种类型对象。出现这种内部错误应该通知用户和妥善终止程序。

Exception 层分成两个分支：一个派生于 RuntimeException，另一个分支包含其他异常。java中重点关注Exception异常层次

  • 由编译错误导致的异常属于 RuntimeException

    • 错误的强制类型转换
    • 数组访问越界
    • 访问 null 指针如果出现 RuntimeException 异常，那么就一定是你的问题

    出现RunTimeException异常，就一定是你的问题（应该在编码时避免 如避免数组下标越界，避免使用空引用）

  • 程序本身没有问题，由于像 I/O 错误这类问题导致的异常属于其他异常

    不是派生于 RuntimeException 类的异常

    • 试图超越文件我末尾继续读取数据
    • 试图打开一个不存在的文件
    • 试图根据给定的字符串查找 Class 对象，而这个字符串表示的类不存在

检查型与非检查型异常

  • 非检查型 (unchecked) 异常，例如越界错误、访问 null 引用

    编译器不希望发生，要求程序员尽量避免触发这类异常

  • 检查型 (checked) 异常

    编译器会检查程序员是否知道这个异常并做好处理异常带来的后果。

非检查型 (unchecked) 异常

  • 派生于 Error 类
  • 派生与 RuntimeException 类

检查型 (checked) 异常

  • 所有其他的异常

声明检查型异常

编译器将检查是否为所有的检查型异常提供了异常处理器

方法不仅需要告诉编译器将要返回什么值，还要告诉编译器有可能发生什么错误

要在方法的首部指出这个方法可能抛出一个异常，以反映这个方法可能抛出的检查型异常

 `public FileInputStream(String name) throws FileNotFoundException`

​ 如果真的抛出了一个异常对象，运行时，系统就会开始搜索知道如何处理 FileNotFoundException 对象的异常处理器

会抛出异常的情况:

  • 调用一个抛出检查型异常的方法，如 FileInputStream 构造器
  • 检测到一个错误，并且利用 throw 语句抛出一个检查型异常
  • 程序出现错误，例如 a[-1] = 0; 会抛出一个非检查型异常 (ArrayIndexOutOfBoundsException)
  • Java 虚拟机或运行时库出现内部错误

前两种情况，必须告诉调用这个方法的程序员可能抛出异常。

异常规范

一个方法可能抛出多个检查型异常类型，在方法首部列出所有的异常类，每个异常类之间用逗号隔开

  • 一个方法必须声明所有可能抛出的检查型异常

  • 非检查型异常

    • 不需要声明 Java 的内部错误，即从 Error 继承的异常

      在我们控制之外

    • 不应该声明从 RuntimeException 继承的非检查型异常

      这种错误在我们控制中，应该从一开始去避免。

方法必须声明所有的检查型异常

子类覆盖超类的一个方法

  • 子类

    方法中声明的检查型异常不能比超类方法中声明的异常更通用

    • 子类可以抛出更特定的异常或者不抛出异常

  • 超类没有抛出任何检查型异常，子类也不能抛出任何检查型异常

抛出的异常实例可能属于声明的异常的类，也可能属于这个类的任意一个子类

如何抛出异常

  • 找到一个合适的异常类

  • 创建这个类的一个对象

  • 将对象抛出

一旦方法抛出了异常，这个方法就不会返回到调用者。

创建异常类

情况：遇到任何标准异常类都无法描述清楚的问题

派生于 Exception 的类或派生于 Exception 的某个子类

两个构造器：一个是默认的构造器，一个是包含详细描述信息的构造器

捕获异常

如果发生异常，但没有在任何地方捕获这个异常，程序就会终止，并在控制台上打印这个异常的类型和一个堆栈轨迹

捕获异常需要设置 try/catch 语句块

3种情况

  1. 如果 try 语句块中任何代码抛出了 catch 子句指定的一个异常类

    程序将跳过 try 语句块的其余代码

    执行 catch 子句中处理器代码

  2. 如果 try 语句块中的代码没有抛出任何异常，那么程序跳过 catch 子句

  3. 如果方法中任何代码抛出了 catch 子句中没有声明的一个异常类型

    方法会立刻退出

对出现异常的处理有两种方法

  1. 将异常传递给调用者

    必须声明会抛出的异常

  2. 捕获并处理异常

一般经验:捕获知道如何处理的异常，继续传播不知道怎样处理的异常

如果编写一个方法覆盖超类的方法，而超类方法中没有抛出异常，则必须捕获方法代码中出现的每一个检查型异常。即不允许子类的 throws 说明符中出现超类方法未列出的异常类

捕获多个异常

对不同类型的异常做出不同的处理为每个异常类型使用一个单独的 catch 子句，在 Java 7 中，同一个 catch 子句中可以捕获多个异常

  • e.getMessage 获得更多信息
  • e.getClass().getName() 异常对象的实际类型

捕获多个异常时，异常变量隐含为 final 变量

捕获多个异常会让代码更简单和高效

再次抛出异常与异常链

  • 没看懂

finally子句

代码抛出一个异常时，就会停止处理方法种剩余的代码。如果这个方法已获得一些只有自己知道的本地资源，这些资源必须被清理。

不管是否有异常被捕获，finally 子句的代码都会执行

finally 子句用于处理资源分配后的清理

try 语句可以只有 finally 子句而没有 catch 子句

3种情况

  1. 代码没有抛出异常，程序首先执行 try 语句块中全部代码，然后执行 finally 子句中的代码，再执行 finally 子句之后的第一条语句

    即 1、2、5、6

  2. 代码抛出异常并在 catch 中捕获

    • try 中执行到语句发生异常的位置，跳过 try 中的剩余部分代码，在一个 catch 子句中捕获。执行 catch 子句中的代码，最后执行 finally 子句中的代码

    • 如果 catch 子句没有抛出异常，则将执行 finally 子句后的第一条语句

      执行顺序为 1、3、4、5、6

    • 如果 catch 子句抛出异常，则异常将抛回这个方法的调用者

    执行顺序为 1、3、5

  3. 代码抛出了一个异常，但没有任何 catch 子句捕获这个异常。程序执行 try 语句块的语句直到抛出异常位置，然后跳过 try 语句块中剩余代码，然后执行 finally 子句的语句，并将异常返回给这个方法的调用者

    执行顺序 1、5

try 语句可以只有 finally 子句而没有 catch 子句

  • 内层 try 语句块时确保关闭输入流
  • 外层 try 语句块时确保报告出现的错误

finally 子句的体，用于清理资源。不要把改变控制流的语句 (return, throw, break, continue) 放在 finally 子句中

try-with-Resources语句(Java7)

try-with-resources 语句 (带资源的 try 语句) 的最简形式

try 块退出时，会自动调用 res.close();

类似python的with语句

try() 中如果只有一个资源，不用分号。如果有多个资源，最后一个资源后面不用分号。

try-with-resources 能够自动清理资源，但是还是要写 throws 或者 catch 语句

使用异常的技巧

  • 异常处理不能代替简单的判断
    • 基本规则：只在异常情况下使用异常
  • 不要过分地细化异常
    • 极端例子：每条语句写一个 try
    • 整个任务放在一个try语句块中
    • 承诺：将正常处理与错误处理分开
  • 充分利用异常的层次结构
    • 应该寻找一个适合的子类或创建自己的异常类
    • 不要为逻辑错误抛出异常
    • 能够将一种异常转换成另一种更加合适的异常，那么不要犹豫。
  • 不要压制异常
    • 异常可能会被忽略
  • 在检测错误时，“苛刻” 比放任更好
  • 不要羞于传递异常

后两个可以总结为“早抛出，晚捕获”

断言

java断言关键字assert

两种形式

  • assert condition
  • assert condition:expression表达式expression唯一目的是产生一个消息字符串。

这两种形式都会计算条件，若结果为false ，则抛出一个AssertionError异常。

第二个语句，表达式将传入AssertionError构造器，并转换为一个消息字符串。

例

默认情况下，断言是禁用的。启用禁用断言是类加载器功能

使用断言的场景

  • 断言失败是致命的、不可恢复的错误
  • 断言检查只是在开发和测试阶段打开
