2021-03-28 effective Java tags Comments

对所有对象通用的方法

Object是一个具体类，但设计它的主要目的是为了扩展。

Object中所有非final方法（equals,hashCode,toString,clone,finalize)都有明确的通用规定（general contract）因为他们被设计成要被覆盖的（override）。

在覆盖这些方法的时候都要遵守这些约定，否则无法与其他依赖于这些约定的类（HashMap,HashSet)一起正常工作。

10覆盖equals时遵守的约定

不覆盖equals方法，类的每一个实例只与它自身相等。

如果满足下列任意条件，就可以不覆盖equals方法

  • 类的每个实例本质上时唯一的

    • 对于代表活动实体而不是值（value）的类符合。如Thread。Object的equals方法就是正确行为

  • 类没有必要提供“逻辑相等”（logical equality）的测试功能

    • 例如java.util.regex.Pattern可以覆盖equals，已检查两个Patterns实例是否代表同一个正则表达式，但设计者不认为用户需要或期望这样的功能。这种情况从Object继承得到的equals即可

  • 超类已经覆盖equals，超类的行为对这个类也合适

    • 例如
      • 多数Set都从AbstractSet继承equals实现，
      • List实现从AbstractList继承equals实现，
      • Map从AbstractMap继承实现

  • 类是私有的，或包级私有。可以确定它的equals方法永远不会被调用

    • 如果非常想规避风险可以覆盖equals方法以确保它不会被调用

何时覆盖equals方法

  • 类具有自己特有的“逻辑相等”（logical equality）概念（不等同于对像等同）
  • 超类没有覆盖equals

通常属于值类情形

值类 仅是一个表示值的类 例如String Integer

此时利用equals方法来比较来比较值的对象是否相时（逻辑上是否相等，而不是是否指向同一个对象）。

不仅需要需要覆盖equals方法，而且这样做也使得这个类的实例可以被用作映射表（map）的键（key），或集合（set）的元素，是映射和集合表现出预期行为

覆盖equals方法的通用规定

来自Object规范

equals方法实现等价关系（equivalence relation）

  • 自反性（reflexive）
  • 对称性（symmetric）
  • 传递性（transitive）
  • 一致性（consistent）
  • 对任意非null的引用值x，x.equals(null);必须返回false

实现高质量euqals诀窍

  1. 使用==操作符检查“参数是否为这个对象的引用”
  2. 使用instanceof操作费检查“参数是否为正确类型”
  3. 把参数转换成正确类型
  4. 对于类中每一个关键域，检查参数中的域是否与该对象对应的域相匹配

注意

  • 覆盖equals总要覆盖hashcode
  • 不要试图让equals方法过于智能
  • 不要将equals声明中的Object 对象替换为其他类型

小结

编写测试equals、hashcode方法十分繁琐，使用Google的开源框架AutoValue框架，自动生成这些方法。（IDE也可实现这样的功能）

总之不要轻易覆盖equals方法。多数情况下，从Object继承的方法即可使用。

覆盖equals一定要比较这个类所有关键域，并查看是否遵守equals的5个条款

11覆盖equals时总要覆盖hashCode

违反这个约定的类，无法结合所有基于散列的集合一起正常工作（HashSet、HashMap）。

Object规范 hashCode约定

  • 应用程序执行期间，对象的equals方法返回的的结果没有被修改，同一个对象调用多次，hashCode方法必须都返回同一个值。在一个应用程序和另一个程序执行过程中hashCode方法返回的值可以不一致
  • 两个对象根据equals(Object)方法比较一致，这两个对象的hashCode产生整数结果必须相同
  • 两个对象根据equals(Object)方法比较不一致，这两个对象的hashCode产生整数结果不必要不相同（但是尽量不同能提高散列表性能

相等的对象必须具有相等的散列码

没有覆盖hashCode而违反的关键约定是第二条：相等的对象必须有相同散列码

散列函数技巧

好的散列函数倾向于“为不同对象产生不相等散列码”（约定3）理想情况下，散列函数应把集合中不相等实例均匀分布到可能的int值上。

  1. 声明一个int变量命名为result，初始化为对象中第一个关键域的散列码c。如步骤2所示
  2. 对象中剩下的每一个关键域f都完成如下
    • 为该域计算interesting类型散列码c
      • 如果该域为基本类型，计算Type.hashCode(f),基本装箱类与f类型相对
      • 该域是一个对象引用，且该类的equals方法通过递归调用equals的方法来比较这个域，则同样为这个域递归调用hashCode。更复杂的比较，则为这个域计算一个范式（canonical representation），然后针对这个范式调用hashCode。如果这个域值为null，返回0（常数)
      • 如果该域是一个数组，需要每一个元素当作单独的域来处理。即利用上述规则，对每个重要元素计算一个散列码。然后根据2.2组合散列码。没有重要元素，可以使用常量（最好不要是0）。如果数组中元素都很重要，可以使用Arrays.hashCode
    • 按照result = 31 * result + c将2.1中计算的散列码c组合到result中
  3. 返回result

完成hashCode编写后一定要注意相等实例是否都有相同散列码

步骤2.2中的乘法部分使得散列值依赖域的顺序，一个类包含多个相似的域，这样的运算会产生一个会产生更好的散列函数

×31 ：31是一个奇素数。如果乘数是偶数，并且乘法溢出，信息会丢失，乘2等价于移位。习惯上使用素数来计算散列结果

31特性： 31*i==(i<<5)-1现代虚拟机自动完成这种优化

按上述方法实例

Object类静态方法，带有任意数量对象，并为他们返回一个散列码。hash

运行速度较慢。因为 传入可变参数引发数组创建，基本类型装箱拆箱

不可变类，可以把散列码缓存在内部。

  • 这类对象多数用做散列键，应在创建实例时计算散列码
  • 否则可选择懒加载

不要试图从散列码中排除一个对象的关键域来提高性能，这样会导致散列函数性能下降（映射到少数散列码）

不要对hashCode方法返回值做成具体的规定，因此客户端无法理所当然的依赖它，这样可以为修改提供灵活性。

小结

覆盖equals必须覆盖hashCode，否则程序无法正常运行。hashCode必须遵守Object规定的通用规约，并完成一定工作。

12始终要覆盖toString

Object提供的toString方法的一个实现。

返回的字符串 类的名称+“@”+散列码的无符号16进制表示法

toString通用规约指出

  • 返回的字符串应该是”简洁但信息丰富，且易于阅读的表达形式“
  • 建议所有子类都覆盖这个方法

好的toString实现可以使类用起来更加舒适，更易于调试。

当对象传递给printf、println、字符串串联符（+）、assert等，toString会自动调用。打印出错信息更清晰

实际使用中，toString方法应返回对象中的所有值得关注信息

使用toString时必须确定是否在文档中指定返回值的格式。。对于值类（电话号码类，矩阵类）建议这么做。

  • 指定格式的好处是，它可以用作一种标准、明确、易读的的对象表示法（CSV）

    如果指定了格式，最好再提供一个相匹配的静态工厂或构造器，便于再对象和字符串表示法之间转换（BigInteger、BigDecimal、多数基本包装类)

  • 不足，如果这个类已经被广泛使用，一旦指定格式，就必须始终如一。

无论指定什么格式（或不指定）

  • 都必须再文档中表明意图
  • 都为toString返回值中包含的所有信息提供一种可通过编程访问的途径

  • 在静态工具类中编写toString没有意义（tip 4）

  • 多数枚举类中不需要编写toString

  • 可以使用AutoValue工具自动生成toString（但是不特定于该类的意义，比如不适合PhoneNumber类）Point类比较适合这种方法

13谨慎覆盖clone

Cloneable接口的目的是作为对象的一个mixin接口（mixin interface）tip20，表明这样的对象可以克隆。

其主要缺陷在于缺少一个clone方法，而Object的clone方法是protected的。如果不借助反射，即使实现Cloneable也不能直接调用clone方法

即使使用反射也不一定保证调用成功，因为不能保证队形一定具有可访问的clone方法。

Cloneable接口决定Object中受保护的clone方法实现的行为：如果一个类实现了Cloneable，Object的clone方法就应该返回该对象的逐域拷贝，否则会抛出CloneNotSupportedException异常。（极端非典型用法）

如何实现一个行为良好的clone方法

实现Cloneable接口的类是为了提供一个功能适当的公有clone方法。（无需构造器就可创建对象）

Object中clone方法的通用约定

  • 创建返回该对象的一个拷贝（拷贝的精确含义取决于该对象的类）一般如下

  • x.clone()!=x结果为true

  • x.clone().getClass()==x.getClass()结果为true

  • x.clone().equals(x)结果为true

  • 按照约定，这个方法应该通过调用super.clone获得。

    如果类及其超类（Object除外）遵守这一约定=>x.clone().getClass()==x.getClass()

  • 返回的对象应该不依赖于被克隆的对象。

    为成功实现这种独立性，可能需要在super.clone之前修改对象的一个或多个域

如果类的clone方法返回的实例不是通过super.clone获得，而是通过构造器获得，编译器就不会发出警告，但是该类的子类调用了super.clone，得到对象就会拥有错误得类，并阻止clone方法得子类正常工作。

如果final类得clone方法没有调用super.clone，这个类就没有必要实现Cloneable接口，因为它不依赖于Object克隆实现行为。

假设在一个类中实现Cloneable接口，且它的超类都提供了行为良好得clone方法。

  • 首先调用super.clone
    • 由此得到原始对象功能完整得克隆

注意

不可变类永远都不应提供clone方法
